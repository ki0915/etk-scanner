# $analysis — 코드 분석 방법론

## 핵심 원칙

패키지 소스를 직접 읽고, 모든 유형의 취약점을 자유롭게 탐색한다.
특정 함수 grep에 의존하지 않는다 — 코드의 의미를 이해하고 버그를 찾는다.

## 분석 흐름

### Step 1: 소스 획득 + 구조 파악
- 소스 다운로드 (git clone)
- 디렉토리 구조 파악 — 핵심 코드 위치, 언어, 의존성
- 코드 규모 측정 (핵심 모듈 및 줄인지)
- README/문서에서 패키지 목적과 사용법 파악

### Step 2: 공격 표면 식별

이 패키지가 받는 **모든 외부 입력**을 나열한다:
- 공개 API 파라미터 (함수 인자, 옵션 객체)
- 파싱 대상 데이터 (파일, 네트워크 데이터, 문자열)
- 환경 변수, 설정 파일
- HTTP 헤더, 쿼리, URL
- 다른 라이브러리로부터 받는 데이터

각 입력에 대해: **현실적으로 공격자가 제어할 수 있는가?**

### Step 3: 코드 통독 + 취약점 탐색

코드를 읽으면서 아래 **모든 카테고리**를 검토한다. 특정 함수명이 아니라 **동작의 의미**를 본다.

#### 메모리/버퍼 안전성 (C/C++/Rust unsafe)
- 버퍼 크기 계산이 올바른가?
- 사용자 제어 값이 인덱스/오프셋/길이로 쓰이는가?
- 정수 오버플로우가 크기 계산에 영향을 주는가?

#### 인젝션
- SQL: 사용자 입력이 쿼리에 직접 삽입되는가?
- Command: shell 실행 시 입력이 이스케이프 없이 사용되는가?
- Path Traversal: 파일 경로 구성 시 `../` 처리가 없는가?
- Template Injection: 템플릿 엔진에 사용자 입력이 들어가는가?

#### 역직렬화
- pickle, yaml.load, eval, exec 등 위험한 역직렬화가 있는가?
- 사용자 제공 데이터가 역직렬화되는가?

#### 레이스 컨디션
- TOCTOU (check-then-use) 패턴이 있는가?
- 파일 시스템 또는 DB에서 읽고-확인-쓰기 순서가 있는가?

#### 인증/권한
- 토큰 검증 로직에 알고리즘 혼동 가능성이 있는가?
- 권한 체크를 우회할 수 있는가?

#### ReDoS
- 정규식이 사용자 입력을 받는가?
- 백트래킹이 폭발적으로 증가할 수 있는 패턴인가?

#### 기타
- Zip Slip (압축 해제 시 경로 검증)
- XXE (XML 외부 엔티티)
- SSRF (서버사이드 요청 위조)

### Step 4: 발견 사항 검증

발견한 취약점에 대해:
1. 실제로 사용자 입력이 해당 코드에 도달할 수 있는가? (공격 경로)
2. 기존 Issue/PR에 이미 보고된 사항인가?
3. 기존 CVE로 등록된 취약점인가?
4. 의도된 동작인가? (문서, 주석 확인)
5. maintainer 입장에서 이 취약점을 거부할 근거가 있는가? (DA 검토)

### Step 5: 결과 기록

Go 판정 시: `00_analysis.md`에 기록 후 `vuln-001/` 폴더 생성
Stop 판정 시: 사유와 함께 `00_analysis.md`에 기록, 종료

## 판정 기준

**GO**: 사용자 입력이 취약한 코드에 도달 가능하고, exploit이 현실적으로 가능
**STOP**: 취약점 코드 존재하나 도달 불가, 이미 알려진 이슈, 의도된 동작
